### ISSUE 7 — Matches N+1 Query: Enrich Server-Side [IT — 🔴 HIGH]

**Problem:** `MatchesPage.tsx:375-414` fires 2 HTTP requests per match result
(Supabase profile fetch + Express goals fetch). 10 matches = 20+ parallel requests.

**Implementation:**

In `src/controllers/matchingController.ts`, after computing match scores, batch-fetch
all profiles and goal nodes in 2 Supabase queries, then join in memory:

```typescript
const userIds = rawMatches.map(m => m.userId);

const { data: profiles } = await supabase
  .from('profiles')
  .select('id, name, avatar_url, bio, current_streak, last_activity_date')
  .in('id', userIds);

const { data: goalNodes } = await supabase
  .from('goal_nodes')
  .select('user_id, name, domain, progress')
  .in('user_id', userIds);

const enriched = rawMatches.map(m => {
  const profile = profiles?.find(p => p.id === m.userId);
  const nodes = goalNodes?.filter(n => n.user_id === m.userId) ?? [];
  const domains = [...new Set(nodes.map(n => n.domain).filter(Boolean))];
  const sharedGoals = nodes
    .sort((a, b) => b.progress - a.progress)
    .slice(0, 3).map(n => n.name);
  const overallProgress = nodes.length
    ? Math.round(nodes.reduce((s, n) => s + n.progress, 0) / nodes.length)
    : undefined;
  return { userId: m.userId, score: m.score, ...profile, domains, sharedGoals, overallProgress };
});

res.json(enriched);
```

Then update `MatchesPage.tsx` to consume the enriched response directly —
remove the entire inner `Promise.all` loop (lines ~375-414).
Result: 1 API call instead of 2N. Page load drops from ~2-4s to <500ms.

### ISSUE 8 — Auth Token: Centralized Axios Interceptor [IT — 🟡 MEDIUM]

**Problem:** Auth token attachment is scattered across dozens of files with manual
`supabase.auth.getSession()` calls. Some components use raw `fetch()`. Silent failures
when session expires.

**Implementation:**

Create `client/src/lib/axiosInstance.ts`:

```typescript
import axios from 'axios';
import { supabase } from './supabase';
import { API_URL } from './api';

const axiosInstance = axios.create({ baseURL: API_URL });

axiosInstance.interceptors.request.use(async (config) => {
  const { data: { session } } = await supabase.auth.getSession();
  if (session?.access_token) {
    config.headers.Authorization = `Bearer ${session.access_token}`;
  }
  return config;
});

axiosInstance.interceptors.response.use(
  res => res,
  async (error) => {
    if (error.response?.status === 401) {
      const { data: { session } } = await supabase.auth.refreshSession();
      if (session) {
        error.config.headers.Authorization = `Bearer ${session.access_token}`;
        return axiosInstance(error.config);
      }
    }
    return Promise.reject(error);
  }
);

export default axiosInstance;
```

Migrate all `axios.get/post/put/delete` calls in protected pages to use
`axiosInstance`. Priority order: `MatchesPage.tsx`, `DashboardPage.tsx`, `ChatRoom.tsx`.
For raw `fetch()` calls in `Navbar.tsx`, either migrate to axiosInstance or create
an `authFetch()` wrapper that attaches the token before calling fetch.

### Session 31 (2026-02-27) — Vercel Routing: Definitive Proxy Fix

**Root Cause (final diagnosis):**
The frontend bundle falls back to `${window.location.origin}/api` when `REACT_APP_API_URL` is empty at build time — meaning all API calls go to the Vercel domain, not Railway directly. Every previous routing attempt failed because:
1. `rewrites` with `/(.*) → /index.html` runs AFTER serverless functions but also AFTER static files — Vercel was serving `index.html` for GET and returning 405 for POST (can't POST to a static file).
2. `routes` with `dest: "/api$1"` pointed back to the local `api/[...slug].ts` serverless function, which ran Express in a cold-start serverless environment with inconsistent behavior.
3. Even with a proxy `rewrite` to Railway, `api/[...slug].ts` still existed → Vercel matched the serverless function BEFORE evaluating rewrites, so the proxy never ran.

**Confirmed working (curl):** Railway backend handles all methods correctly — `POST /api/groups`, `POST /api/goals`, `POST /api/posts` all return 2xx with real UUIDs.

**Fixes Applied (commits `08dee38`, `6a01e88`):**

1. **`vercel.json`** — switched to proxy `rewrites`:
   ```json
   "rewrites": [
     { "source": "/api/:path*", "destination": "https://web-production-646a4.up.railway.app/api/:path*" },
     { "source": "/(.*)", "destination": "/index.html" }
   ]
   ```
   `headers` block handles static asset caching separately. This makes Vercel a transparent reverse proxy — no env var needed in the frontend bundle.

2. **Deleted `api/[...slug].ts` and `api/index.ts`** — removed all Vercel serverless functions. Without them, no function can intercept `/api/*` before the proxy rewrite runs.

3. **`client/src/features/chat/ChatRoom.tsx` line 139** — guarded `setMessages` against non-array:
   ```typescript
   // Before: setMessages(response.data || []);
   setMessages(Array.isArray(response.data) ? response.data : []);
   ```
   Prevents `I.map is not a function` crash if API returns unexpected data.

**Architecture going forward:**
- Vercel serves only the static React build + proxies `/api/*` to Railway.
- `REACT_APP_API_URL` env var is no longer needed (can be removed from Vercel dashboard if set).
- Railway remains the sole Express runtime.

**Final fix (commit `1b07334`) — the actual root cause:**
The real problem was never Vercel routing. `client/src/lib/api.ts` had a production fallback of `${window.location.origin}/api` — so when `REACT_APP_API_URL` wasn't baked into the bundle, every API call hit Vercel (which served `index.html` for GET and 405 for POST). Fix: replaced the fallback with `'https://web-production-646a4.up.railway.app/api'` hardcoded. Frontend now always calls Railway directly in production. **Confirmed working by user.**

## Product Strategy — Monetization & Retention

### 💰 Monetization Model

**Tier 1 — Free**
- Limited active goals (max 5)
- Basic goal tracking & check-ins
- Limited AI matches per month

**Tier 2 — Pro**
- Unlimited goals
- Advanced analytics on progress
- Unlimited AI-assisted matching
- Personalized AI coaching (Master Roshi, unlimited sessions)

**Tier 3 — Premium / Team**
- Group accountability rooms
- Team reporting & metrics
- Custom matching algorithms
- Admin dashboards

**Stripe is already wired in.** Priority: enforce gates consistently server-side. Single source of truth for tier → features mapping. Gate Master Roshi behind Pro (costs real API money). Gate goal tree nodes (Free = 5 max, Pro = unlimited — half-done with edit count gate).

**Paid Matching Credits** — alternative to subscription: 1 credit = 1 AI-matched accountability session. Bulk discounts. Only viable if matching feels scarce and high-value.

**AI Premium Insights (add-ons or Pro tier)**
- Personalized AI check-in summaries
- NLP sentiment tracking over goals/time
- AI-generated weekly progress reviews

**Coaching Marketplace Commission** — platform takes % on coach bookings. Complex (needs Stripe Connect). Build marketplace UI first, add payments once supply-side exists.

**Corporate / Team Licensing** — sell to companies. Admin dashboards, engagement reporting, private matching within org. Highest ACV, needs sales motion. Offer "Contact us" CTA until 10+ individual Pro users.

**Skip for now:** virtual goods/badges, Notion/Slack integrations, NLP sentiment (needs data volume).

**Priority order:** 1) enforce Pro gates → 2) gate Master Roshi → 3) coaching marketplace commission → 4) matching credits → 5) corporate licensing.

---

### 🧠 Engagement & Retention Design (Hooked Model)

**Goal: make Praxis as compulsive as Instagram for goal-oriented people.**

#### Triggers
- "Your accountability partner just checked in."
- "You're 1 day away from losing your streak."
- "3 people matched your goal profile."
- "You haven't updated your goal in 48 hours."
- → Push notifications + email nudges. Streak warnings. Light FOMO around partner responsiveness.

#### Fast, Frictionless Actions
Micro-actions must be < 3 seconds:
- Check in (1 click)
- Send accountability nudge
- Mark task complete
- React to partner update
- If it feels like work, it won't be habitual.

#### Variable Rewards (the core hook)
Uncertainty = dopamine. Don't guarantee the reward every time:
- Unexpected partner messages
- Random AI insights from Master Roshi
- Surprise streak boosts
- Weekly "compatibility score" changes
- Unlockable accountability types

#### Investment / Lock-in
The more history, the harder it is to leave:
- Long progress histories with visual streak graphs
- Partner relationship strength tracking
- Goal evolution over time
- The data IS the product — users don't want to lose it

---

### 🎮 Specific Addictive Mechanics to Build

**Social Streaks** — don't just show "7 day streak", show "You and Alex have a 7-day mutual accountability streak." Breaking it hurts both people. Emotional = sticky.

**Social Status Layers**
- Commitment score, Reliability score, Discipline percentile
- "You're in the top 12% of consistent users."
- Elite accountability tiers
- Status is addictive.

**Dynamic Progress Visualization**
- Growing progress bars, animated weekly summaries
- AI narrative recap: "This week you showed exceptional consistency."
- Gamified dashboards increase return visits.

**AI as Parasocial Companion (Master Roshi)**
- Daily messages: "You've been slipping.", "Today matters.", "You're outperforming 82% of users."
- The AI becomes a presence, not a tool.

**Micro-Drama Between Partners**
- Response timers
- "Seen but not checked in"
- Gentle call-outs
- Rematch tension
- Weekly compatibility scoring
- Light social pressure = very sticky.

CREATE TABLE IF NOT EXISTS public.notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  type TEXT NOT NULL, title TEXT NOT NULL, body TEXT, link TEXT,
  read BOOLEAN DEFAULT false,
  actor_id UUID REFERENCES profiles(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_notifications_user ON public.notifications(user_id, created_at DESC);
ALTER TABLE public.notifications ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users manage own notifications" ON public.notifications FOR ALL USING (auth.uid() = user_id);

CREATE TABLE IF NOT EXISTS public.mutes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  muted_user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  muted_room_id UUID REFERENCES chat_rooms(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ DEFAULT now()
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_mutes_user ON public.mutes(user_id, muted_user_id) WHERE muted_user_id IS NOT NULL;
CREATE UNIQUE INDEX IF NOT EXISTS idx_mutes_room ON public.mutes(user_id, muted_room_id) WHERE muted_room_id IS NOT NULL;
ALTER TABLE public.mutes ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users manage own mutes" ON public.mutes FOR ALL USING (auth.uid() = user_id);

## Session 52 (2026-03-07) — P2P Duels / Challenges Feed

### Overview
Full P2P duel system: users can bet PP against each other on goal challenges, with an open feed sorted by matching relevance.

### New: duels table (§31 migration)
```sql
CREATE TABLE IF NOT EXISTS public.duels (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  creator_id UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  opponent_id UUID REFERENCES public.profiles(id) ON DELETE SET NULL,
  title TEXT NOT NULL,
  description TEXT,
  category TEXT NOT NULL,
  stake_pp INT NOT NULL DEFAULT 50,
  deadline_days INT NOT NULL DEFAULT 7,
  deadline DATE NOT NULL,
  status TEXT NOT NULL DEFAULT 'open',
  won_by UUID REFERENCES public.profiles(id) ON DELETE SET NULL,
  creator_claimed BOOLEAN NOT NULL DEFAULT false,
  opponent_claimed BOOLEAN NOT NULL DEFAULT false,
  creator_claimed_at TIMESTAMPTZ,
  opponent_claimed_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- RLS: SELECT USING (true), INSERT WITH CHECK creator, UPDATE USING creator OR opponent
```
**Must run §31 in Supabase SQL Editor.**

### Backend: duelController.ts + duelRoutes.ts
- `listDuels`: fetches open duels, computes per-duel relevance score (50% domain match vs viewer's goal tree + 30% freshness decay `e^(-hours/72)` + 20% stake interest), sorts descending
- `myDuels`: returns all duels where user is creator or opponent
- `createDuel`: deducts stake PP (blocks if insufficient), sets status `open` (public) or `pending` (direct @username)
- `acceptDuel`: opponent deducts their stake, status → active; push notification to creator
- `declineDuel`: status → declined, full refund to creator
- `cancelDuel`: creator-only before acceptance, full refund
- `claimDuel`: marks `creator_claimed` / `opponent_claimed`; when both set, tiebreaks by `claimed_at` timestamp (first claimer wins)
- `concedeDuel`: immediate opponent win
- `resolveDuel` + `awardWinner`: winner gets `2 × stake × 0.95` (5% house fee); both parties get push notification
- Routes registered at `GET /api/duels`, `GET /api/duels/mine`, `POST /api/duels`, `POST /api/duels/:id/{accept,decline,cancel,claim,concede}`

### Frontend: ChallengesPage.tsx
- **Open Feed tab**: category filter chips (10 domains, color-coded), DuelCard grid sorted by relevance
- **My Duels tab**: groups by pending / active / history
- **DuelCard**: status chip, stake/pot/winnings display, VS layout (creator avatar ↔ opponent or "Open slot"), claim indicators, contextual action buttons:
  - Open + not creator → "Accept (N PP)"
  - Creator + open/pending → "Cancel"
  - Opponent + pending → "Accept" + "Decline"
  - Active + unclaimed → "I won!" + "Concede"
